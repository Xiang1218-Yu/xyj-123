# 全局状态管理设计分析报告

## 一、技术选型概览

### 1.1 核心技术栈

| 技术 | 版本 | 用途 |
|------|------|------|
| Zustand | v5.0.3 | 全局状态管理 |
| persist 中间件 | - | 状态持久化（localStorage） |
| React useState | 18.3.1 | 组件局部状态 |
| React Router | v7.3.0 | 路由参数通信 |

### 1.2 整体架构

项目采用 **单一全局 Store + 组件局部状态** 的混合模式：

- **全局状态**：所有业务数据统一存储在 Zustand Store 中
- **局部状态**：UI 交互状态（如展开/折叠、搜索框、Tab 切换等）使用 `useState` 管理
- **无 React Context**：项目未使用 `createContext` / `useContext` 进行状态传递

## 二、Store 设计详解

### 2.1 Store 位置与结构

- 入口文件：[store/index.ts](file:///Users/tog/Desktop/code/solo/xyj-123/src/store/index.ts)
- 约 1600 行代码，单文件集中式管理
- 使用 `create` + `persist` 中间件创建

```typescript
export const useAppStore = create<AppState>()(
  persist(
    (set, get) => ({
      // 状态数据...
      // Action 方法...
    })
  )
);
```

### 2.2 状态实体清单

Store 中共管理 **30+ 种数据实体**，按业务域可分为：

| 业务域 | 实体 |
|--------|------|
| 宠物与主人 | `owners`, `pets`, `petBreeds`, `breedArticles`, `favoriteBreeds` |
| 仪式与火化 | `ceremonies`, `ceremonyTemplates`, `cremations`, `furnaces`, `furnaceMaintenances`, `furnaceProcesses` |
| 骨灰与纪念 | `urns`, `memorialRecords`, `albums`, `photos`, `petLifeStories` |
| 套餐与合同 | `serviceItems`, `funeralPackages`, `contractTemplates`, `contracts`, `contractSignatures`, `contractTimelines` |
| 员工管理 | `employees`, `shiftSchedules`, `leaveRequests`, `attendanceRecords` |
| 商城与订单 | `memorialProducts`, `cartItems`, `memorialOrders` |
| 预约管理 | `reminders`, `timeSlotLocks`, `appointmentChangeLogs` |

### 2.3 Action 设计模式

每个实体通常包含 **CRUD + 查询** 的标准方法集，以 `owners` 为例：

```typescript
// 新增
addOwner: (owner: Omit<Owner, 'id'>) => Owner;
// 更新
updateOwner: (id: string, data: Partial<Owner>) => void;
// 删除
deleteOwner: (id: string) => void;
// 单条查询
getOwnerById: (id: string) => Owner | undefined;
```

**高级业务 Action** 示例：

- `calculatePackagePrice` - 套餐价格计算
- `generateContractNo` - 合同编号生成
- `approveLeaveRequest` / `rejectLeaveRequest` - 请假审批
- `advanceProcessStage` - 火化流程推进
- `addToCart` / `createOrder` - 购物车与下单
- `archiveSignedContractsAutomatically` - 合同自动归档

### 2.4 ID 生成策略

使用统一的 `generateId` 函数生成分布式 ID：

```typescript
const generateId = (prefix: string) =>
  `${prefix}-${Date.now()}-${Math.random().toString(36).substring(2, 9)}`;
```

特点：
- 带业务前缀（如 `pet-`, `ceremony-`, `order-`）
- 基于时间戳 + 随机数
- 客户端生成，无需服务端

### 2.5 持久化配置

使用 `persist` 中间件将整个 Store 持久化到 `localStorage`：

- **优点**：刷新页面数据不丢失，适合 demo/原型阶段
- **注意**：所有状态都会被持久化，包括购物车、表单临时状态等

## 三、组件间通信方式

### 3.1 通信方式汇总

| 通信方式 | 适用场景 | 典型示例 |
|----------|----------|----------|
| Zustand 全局 Store | 跨页面、跨组件共享业务数据 | 宠物列表、购物车、预约数据 |
| Props 传递 | 父子组件数据传递 | [PageHeader](file:///Users/tog/Desktop/code/solo/xyj-123/src/components/PageHeader.tsx) 的 title/actions |
| useState 局部状态 | 单组件内 UI 状态 | 搜索框、Tab 切换、展开/折叠 |
| 路由参数 | 页面间标识传递 | `useParams` 获取宠物 ID |
| 导航跳转 | 页面间流转 | `useNavigate` 编程式导航 |

### 3.2 全局 Store 使用模式

在 36 个文件中共有 129 处 `useAppStore` 调用，是最主要的状态获取方式。

**方式一：解构多个状态**

```typescript
// PetList.tsx
const { pets, owners } = useAppStore();
```

**方式二：选择器精确获取**

```typescript
// MemorialShop.tsx
const memorialProducts = useAppStore((s) => s.memorialProducts);
const cartItems = useAppStore((s) => s.cartItems);
const addToCart = useAppStore((s) => s.addToCart);
```

> **注意**：选择器写法可以减少不必要的重渲染，但项目中两种写法并存。

### 3.3 父子组件通信

以 [Timeline](file:///Users/tog/Desktop/code/solo/xyj-123/src/components/Timeline.tsx) 组件为例：

```typescript
// 父组件传入数据
<Timeline nodes={story.nodes} compact={false} />

// 子组件通过 Props 接收
interface TimelineProps {
  nodes: StoryNode[];
  compact?: boolean;
}
```

**特点**：
- 纯展示组件，无内部状态
- 数据完全由父组件控制
- 通过 Props 回调模式在项目中较少见

### 3.4 页面级状态管理

以 [PetDetail.tsx](file:///Users/tog/Desktop/code/solo/xyj-123/src/pages/PetDetail.tsx) 为例：

```typescript
// 全局状态：业务数据
const { getPetById, getOwnerById, ceremonies, cremations, urns, reminders, deletePet } = useAppStore();

// 局部状态：UI 交互
const [activeTab, setActiveTab] = useState<TabType>('basic');

// 路由参数：页面标识
const { id } = useParams<{ id: string }>();
```

**状态分层清晰**：
- 业务数据 → 全局 Store
- UI 交互状态 → 组件内部 useState
- 页面标识 → 路由参数

## 四、状态提升策略

### 4.1 提升原则

项目遵循以下状态提升原则：

| 状态类型 | 存放位置 | 原因 |
|----------|----------|------|
| 业务实体数据 | Zustand Store | 需跨页面共享、需持久化 |
| 表单数据 | 组件内 useState | 仅表单页使用，提交后同步到 Store |
| UI 交互状态 | 组件内 useState | 仅当前组件关心，无需共享 |
| 搜索/筛选状态 | 组件内 useState | 页面级交互，刷新后重置 |
| 计算派生数据 | useMemo 本地计算 | 避免重复计算，不存储冗余 |

### 4.2 典型场景分析

**场景一：表单数据处理**（以 [CeremonyForm](file:///Users/tog/Desktop/code/solo/xyj-123/src/pages/CeremonyForm.tsx) 为例）

```
表单编辑过程 → 局部 useState (formData)
     ↓
用户提交 → 调用 Store Action (addCeremony / updateCeremony)
     ↓
Store 更新 → 全局状态同步 → 其他页面自动刷新
```

**场景二：购物车状态**

```
商品列表页 → addToCart → Store 更新
     ↓
购物车抽屉 → 从 Store 读取 cartItems → 实时展示
     ↓
结算页 → 从 Store 读取 → createOrder → 清空购物车
```

**场景三：搜索筛选**

```
搜索关键词 → 局部 useState (searchQuery)
     ↓
useMemo 过滤 → filteredPets
     ↓
列表渲染
```

> 搜索状态不提升到全局，刷新后重置，符合用户预期。

### 4.3 派生数据计算策略

大量使用 `useMemo` 在组件内计算派生数据，避免在 Store 中存储冗余：

```typescript
// Dashboard.tsx
const stats = useMemo(() => {
  const todayCeremonies = ceremonies.filter(/* ... */);
  const pendingCremations = cremations.filter(/* ... */);
  return { petCount, todayCeremonyCount, /* ... */ };
}, [pets, ceremonies, cremations, urns]);
```

Store 中也提供查询方法，但以函数形式存在，不缓存结果：

```typescript
getUrnByPetId: (petId) => get().urns.find((u) => u.petId === petId && u.status === 'stored'),
getAlbumsByPetId: (petId) => get().albums.filter((a) => a.petId === petId),
```

## 五、设计优点

### 5.1 开发效率高

- **API 极简**：Zustand 学习成本低，几行代码即可创建 Store
- **模板统一**：每个实体的 CRUD 模式一致，易于扩展新实体
- **无需 Provider**：直接 import hook 即可使用，减少组件嵌套

### 5.2 状态集中管理

- 所有业务数据在一个 Store 中，便于查找和维护
- 跨模块数据操作方便（如删除员工同时清理排班/考勤/请假数据）
- 整体状态可通过 devtools 一次性查看

### 5.3 持久化开箱即用

- `persist` 中间件一行配置实现 localStorage 持久化
- 适合原型开发和 demo 展示，刷新页面数据不丢失

### 5.4 状态分层清晰

- 业务数据与 UI 状态分离明确
- 全局状态和局部状态边界清晰
- 避免了"全部放全局"的反模式

### 5.5 性能尚可

- Zustand 基于选择器的订阅机制，只订阅用到的状态
- 大量使用 `useMemo` 避免重复计算
- 没有使用 Context，避免了 Provider value 变化导致的全量重渲染

## 六、存在的问题与不足

### 6.1 单文件 Store 过于庞大

**问题**：[store/index.ts](file:///Users/tog/Desktop/code/solo/xyj-123/src/store/index.ts) 约 1600 行，包含 30+ 实体和 80+ 方法。

**影响**：
- 可读性差，难以快速定位
- 多人协作容易冲突
- 类型定义和实现混在一起

**优化建议**：按业务域拆分 Store 或使用 slices 模式。

### 6.2 全量持久化可能有问题

**问题**：`persist` 中间件默认持久化整个 Store，包括临时状态。

**潜在风险**：
- 大量数据序列化/反序列化影响性能
- 某些敏感数据不应持久化
- 版本升级后数据结构变化可能导致错误

**优化建议**：配置 `partialize` 只持久化必要数据。

### 6.3 缺少状态选择器优化

**问题**：部分组件直接解构整个 Store，而非使用选择器。

```typescript
// 可能导致不必要的重渲染
const { pets, owners } = useAppStore();
```

**优化建议**：
- 统一使用选择器写法
- 配合 `shallow` 比较优化多字段选择

### 6.4 缺少中间件分层

**问题**：业务逻辑直接写在 Action 中，缺少日志、错误处理、乐观更新等横切关注点。

**优化建议**：
- 添加 `devtools` 中间件便于调试
- 封装统一的错误处理模式
- 考虑引入 Immer 简化不可变更新

### 6.5 无错误边界和加载状态

**问题**：Store 中没有 loading / error 状态，所有操作都是同步的。

**影响**：
- 接入真实 API 后需要大量改造
- 无法乐观更新和处理失败回滚

### 6.6 主题状态独立管理

**问题**：[useTheme.ts](file:///Users/tog/Desktop/code/solo/xyj-123/src/hooks/useTheme.ts) 单独使用 `useState + localStorage`，未纳入 Zustand。

**影响**：
- 状态管理模式不统一
- 如需在其他地方访问主题状态不方便

**优化建议**：将主题状态也纳入 Zustand Store。

## 七、架构演进建议

### 7.1 短期优化（1-2 天）

1. **Store 模块化拆分**：按业务域拆分为多个 slice，再合并
2. **持久化白名单**：只持久化核心业务数据
3. **统一选择器写法**：所有组件使用选择器 + shallow
4. **添加 devtools**：开发环境启用 Redux DevTools

### 7.2 中期优化（1 周）

1. **引入 Immer**：简化状态更新写法
2. **添加错误处理**：统一的 try-catch 和错误状态
3. **主题状态迁移**：useTheme 合并到 Zustand
4. **抽象通用 CRUD**：减少重复模板代码

### 7.3 长期演进（接入真实后端）

1. **引入 React Query / SWR**：服务端状态管理
2. **Zustand 只存客户端状态**：UI 状态、用户偏好等
3. **乐观更新与缓存策略**：提升用户体验
4. **状态机管理复杂流程**：如火化流程、合同签署流程

## 八、总结

### 8.1 设计评价

本项目的状态管理设计**整体合理，适合当前阶段**：

- ✅ 技术选型恰当，Zustand 轻量高效
- ✅ 状态分层清晰，全局与局部边界明确
- ✅ 组件通信方式多样且使用得当
- ✅ 状态提升策略符合 React 最佳实践
- ⚠️ 单文件 Store 过大，需要模块化拆分
- ⚠️ 缺少优化手段，性能和可维护性有提升空间

### 8.2 适用场景

当前架构特别适合：

- 小型到中型项目（20-50 个页面）
- 原型快速开发
- 以 CRUD 为主的管理后台
- 团队规模较小（1-3 人前端）

当项目规模扩大或接入真实 API 后，建议按第七章的演进路线逐步优化。
